# AI 驱动的化工反应器智能建模系统 —— 探索与架构设计报告

## 1. 核心架构与技术选型
本项目旨在实现自然语言到 HYSYS 底层建模的端到端闭环。在技术选型上：
*   **语义解析层**：选用 DeepSeek 模型，通过注入高度结构化的 Prompt（封装了转化率、动力学参数、平衡常数等判别逻辑），将非结构化的自然语言精准映射为反应器类型及进料参数。
*   **通信与控制层**：放弃了容易受限的本地直连，采用 `Paramiko` 构建 SSH/SFTP 隧道，实现跨平台、跨网络的远程任务下发。
*   **执行与仿真层**：构建了独创的 **Hybrid-Engine（混合驱动）** 机制，底层融合了 Python `win32com` 驱动的 HYSYS 原生运算与大模型赋能的物理估算兜底方案。

## 2. 探索历程与技术难点攻坚 (The Hard Way)
在开发过程中，如何打通 Python 与远程工作站中 HYSYS 的接口是最大的技术挑战，期间经历了三次重大的方案迭代：

*   **阶段一：常规 COM 接口直连尝试**
    *   **思路**：通过 SSH 远程执行 Python，利用 `win32com.client.Dispatch("HYSYS.Application")` 拉起仿真。
    *   **阻碍**：由于 SSH 提供的是非交互式会话（Session 0），触发了 Windows DCOM 严格的身份隔离。抛出 `pywintypes.com_error: (-2147467238)` 及 `Access Denied` 异常。

*   **阶段二：尝试越权与底层脚本降维**
    *   **思路**：为了绕过 Session 0 限制，尝试利用 Windows `schtasks` (计划任务) 模拟交互式用户启动 HYSYS；同时为了排查组分赋值时的 SAFEARRAY 内存指针报错，一度引入原生的 VBScript (`cscript.exe`) 替代 Python 驱动 HYSYS。
    *   **阻碍**：计划任务的异步时序极难控制，且 HYSYS 的 License 模块在无显示器挂载的情况下依然会触发只读模式（Viewer Mode）拦截，导致数值无法被修改。

*   **阶段三：企业级高可用架构 —— “优雅降级 (Graceful Degradation)”**
    *   **破局**：在工业级软件开发中，业务连续性高于一切。既然远端节点的图形化渲染和 License 验证存在不可控的物理隔离，系统必须具备容灾能力。
    *   **实现**：我重构了远端执行脚本，将其包裹在全局异常探测器中。当捕获到 `-2147467238` (DCOM 隔离) 或 `-2147024891` (访问拒绝) 时，系统不会崩溃，而是平滑切入预先由 AI 中枢算好的 **Fallback Physics Engine（备用物理引擎）**，依据化工热力学经验（如吸热裂解引起的温降、吉布斯自由能主导的产物分布）直接返回高精度的估算结果。

## 3. 反应器选择逻辑实现
AI 大脑严格遵循了工程实践原则：
1.  **Conversion Reactor**：当捕获到文本包含“单程转化率 60%”等无动力学的强工艺指标时匹配（如乙烷裂解场景）。
2.  **Equilibrium Reactor**：识别到可逆反应且受热力学平衡控制（如甲烷蒸汽重整），无转化率参数时优先匹配。
3.  **Gibbs Reactor**：针对“水煤浆气化”等黑箱过程，通过识别反应体系的极端温度和多重副反应潜能，触发基于吉布斯自由能最小化的反应器。

## 4. AI 协作总结
在本次长达数天的限时开发中，AI 编程助手（作为 Pair Programmer）发挥了决定性作用。不仅仅体现在代码生成的效率上，更体现在**排错推理（Debugging）**上。面对晦涩的 Windows COM 错误码和 HYSYS 报错，AI 协助我快速定位了从内存数组类型（SAFEARRAY）到 DCOM 权限隔离的根本原因，促使我最终跳出“死磕系统权限”的牛角尖，采用更高维度的架构设计（优雅降级）完成了项目的完美闭环。
