# 数据胶囊大载荷外置 — 实验报告

## 1. 实验目的

理解当数据胶囊中 **`Data` 体积极大**（例如 1GB）时，传统「单 JSON 内联载荷」方式存在的问题；掌握通过 **清单（manifest）+ 外置二进制文件 / 流式传输** 的方式规避问题，并完成 **应用 A / 应用 B** 在 **文件目录** 与 **TCP 网络** 两种形态下的互通与解析验证。

## 3. 任务问题回答

### 问题 1：当 `Data` 的 value 很大（例如 1GB）时，会出现什么问题？

若仍将整块业务数据放在 JSON 的 **`Data`** 字段中（常见为 Base64 等文本编码），通常会遇到：

1. **体积膨胀**：Base64 约为原始字节的 4/3，单文档更大，传输与存储成本更高。  
2. **内存压力**：不少实现会一次性将 JSON 读入内存并解析，超大字符串易导致 **OOM** 或严重换页。  
3. **解析与拷贝开销**：解码、转义处理、中间字符串拷贝会使 CPU 与内存峰值进一步恶化，**解析时间**不可接受。  
4. **难以流式处理**：整包 JSON 不利于分块读取、断点续传与边下边验。  
5. **签名与校验**：若实现不当，验签或哈希可能对「整段 `Data`」做非流式拼接，同样触发 **大缓冲** 问题。

### 问题 2：如何解决？本实验给出的具体实例是什么？

**思路**：将 **大体量字节** 移出 JSON，JSON 仅保留 **元数据**、**密钥封装等字段的占位或真实值**（本实验为 mock）、以及 **对外置载荷的引用**（路径、长度、可选 SHA256）；传输时采用 **流式读写 / 流式收发**，用 **长度 + 哈希** 保证与清单一致。

**本仓库中的具体实例**：

- **清单文件** `capsule.manifest.json`：顶层字段与课堂「数据胶囊」形态对齐（`MetaData`、`EncryptedDataKey`、`Data`、`Signature`），其中 **`Data` 为 `null`**，大文件由 **`DataRef`**（`external_file`、相对路径、`size_bytes`、`sha256_hex`）描述。  
- **外置载荷** `payload.bin`：由 A 按指定大小 **流式生成**（常数级内存块循环写入），并计算 SHA256 写入清单。  
- **应用 A**：生成 `payload.bin` 与清单；`--mode file` 写入目录，`--mode tcp` 按自定义帧将「清单字节 + 载荷字节」发给 B。  
- **应用 B**：`--mode file` 读取目录内清单与引用文件并校验大小与哈希；`--mode tcp` 监听连接，收满 `size_bytes` 后落盘并可选比对 `sha256_hex`。  
- **签名与 EDK**：按任务要求使用 **mock**（占位字符串，不做真实密码学验签与解密），仅保证 **结构完整性与载荷完整性（SHA256）** 可验证。

## 4. 程序任务：设计思路与实现说明

### 4.1 总体设计

采用 **「小 JSON + 大文件」** 的逻辑胶囊形态：逻辑上仍是一份胶囊（元数据 + 对数据的描述 + 签名字段），物理上由 **一个 JSON 文件** 与 **一个（或一条流上的）二进制载荷** 组成，避免 1GB 数据进入 JSON。

```text
┌─────────────┐     写目录 或  TCP 帧      ┌─────────────┐
│  Application A  │ ────────────────────► │  Application B  │
│  (app_a.py)     │                       │  (app_b.py)     │
└─────────────┘                       └─────────────┘
       │                                       │
       ▼                                       ▼
capsule.manifest.json                    解析清单 + 读/收 payload.bin
payload.bin
```

### 4.2 模块划分

| 模块 | 职责 |
|------|------|
| **`capsule_format.py`** | 定义 `DataRef`；`default_lab_metadata()` 生成与参考胶囊相近的 `MetaData`；`build_manifest()` 组装顶层 JSON；`parse_manifest_json_bytes()` / `read_manifest()` 校验并解析；兼容早期带 `format_version` 的清单。 |
| **`payload_io.py`** | `parse_size()` 解析 `1M`/`1G` 等；`stream_write_payload()` **分块**写确定性伪随机载荷并累计 SHA256；`stream_copy_and_hash()` / `stream_from_socket_to_file()` 分块读或从 socket 收并哈希；`send_all()` / `recv_exact()` 保证收发完整。 |
| **`app_a.py`** | CLI：生成载荷与清单；file 模式写 `out-dir`；tcp 模式先写临时文件再 **分块读取发送** 载荷。 |
| **`app_b.py`** | CLI：file 模式校验路径、大小、哈希；tcp 模式 **先收 8 字节大端长度 + 清单**，再按 `DataRef.size_bytes` **流式**写入 `received/payload.bin` 等。 |

### 4.3 TCP 传输帧格式（应用 A → 应用 B）

为在一条 TCP 连接上顺序传递「清单 + 二进制」，采用简单定界：

1. **8 字节无符号大端整数**：后续 manifest 的字节长度。  
2. **manifest 原始 UTF-8 字节**（即 `capsule.manifest.json` 的文件内容）。  
3. **payload 原始字节**，长度严格等于清单中 `DataRef.size_bytes`。

B 端对 manifest 长度设置上限（如 64MB），防止畸形头攻击。

### 4.4 清单与参考结构的对应关系

- 与参考胶囊一致：`MetaData`、`EncryptedDataKey`、`Data`、`Signature`。  
- **扩展**：增加 **`DataRef`**，并在本实验中约定 **`Data == null`** 表示载荷外置。  
- **`EncryptedDataKey` / `Signature`**：实验用占位（mock），不实现真实加解密与验签。

## 5. 实验步骤（PowerShell 示例）

在仓库根目录执行：

```powershell
# 1G 载荷 + 写入 ship 目录
python app_a.py --mode file --size 1G --out-dir ship

# B 端读取 ship 并校验
python app_b.py --mode file --in-dir ship
```

TCP 模式需两个终端：先启动 B 监听，再在另一终端运行 A 连接（端口默认 `9000`，可用 `--port` 修改）。

## 8. 总结

本实验通过 **外置 `payload.bin` + 清单中的 `DataRef` 与 SHA256`**，避免了超大 `Data` 内联 JSON 带来的体积、内存与解析问题；通过 **应用 A / B** 与 **文件 / TCP** 两种路径验证了 **结构可解析** 与 **载荷完整性**；密码学相关字段按任务要求 **mock**，真实环境需替换为 **真实 EDK、密文 `Data` 或等价外置策略、以及可验证签名**。
